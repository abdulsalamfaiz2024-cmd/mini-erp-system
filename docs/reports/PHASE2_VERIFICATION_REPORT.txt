4. FAILURE SCENARIO ANALYSIS
----------------------------
1. **Two approvers simultaneously**:
   - Both read `current_state`. Both validate. Both insert `ApprovalRecord`.
   - The `DocumentStateTracker` update might overwrite each other.
   - *Risk*: Multiple approval records for same state possible.
   - *Mitigation*: `UNIQUE` constraint on tracker prevents duplicate active states, but not duplicate history.

2. **Insufficient Approval Level**:
   - `ApprovalService.approve_document` raises `PermissionError` (Line 116).
   - *Confirmed by Test*: `test_workflow_engine.py` scenario [5].

3. **Skip Workflow States**:
   - Impossible via public API. `get_next_state` (Line 89) strictly requires `sequence_order > current`.
   - `DocumentStateTracker` prevents "jumping".

4. **Approve Already-Approved Doc**:
   - If state advanced, `current_state_id` changes. Old approval call would fail validation or apply to new state?
   - Code fetches *current* state at runtime. If advanced, it approves the *next* state (if user has permission).

6. IDENTIFIED RISKS
-------------------
1.  **Missing Integration**: The engine sits in isolation. Sales orders still use the old logic (`status = 'approved'` string manipulation) until integrated.
2.  **Concurrency**: Lack of explicit row locking (`SELECT FOR UPDATE` is tricky in SQLite) means potential for double-approval race conditions under high load.
3.  **Schema Violation**: `ON DELETE CASCADE` on `workflow_states` violates the strict "No Cascading Deletes" rule (Risk: Accidental workflow deletion wipes history).
