Description: Services update BOTH workflow state AND legacy status field

Occurrences:
  1. sales_orders.status (synced with workflow)
  2. sales_returns.status (manual update)
  3. issue_orders.status (manual update)
  4. expenses.status (manual update)
  5. journal_entries.is_posted (manual update)
  6. assets.status (NO WORKFLOW)

Risk Assessment:
  - Sales Orders: LOW (sync mechanism in place)
  - All others: MEDIUM (potential drift)
  - Assets: HIGH (no workflow at all)

Recommendation:
  Either:
    A) Remove status columns, use workflow state queries only
    B) Enforce status updates ONLY via workflow triggers/hooks

ENFORCEMENT RECOMMENDATIONS
═══════════════════════════════════════════════════════════════════

1. DATABASE CONSTRAINTS
   Add foreign key constraint:
   ALTER TABLE document_state_tracker 
   ADD CONSTRAINT fk_workflow_state 
   FOREIGN KEY (current_state_id) REFERENCES workflow_states(state_id)
   
   This prevents invalid workflow states.

2. SERVICE LAYER VALIDATION
   All services MUST call WorkflowService.initialize_document_state
   before creating any document.
   
   Recommended: Create decorator @requires_workflow(doc_type)

3. STATUS FIELD DEPRECATION
   Phase out manual status fields:
   - Mark status columns as deprecated
   - Create views that derive status from workflow
   - Remove status columns in v2.0

4. WORKFLOW HOOK SYSTEM
   Implement workflow event hooks:
   - on_state_enter(doc_type, doc_id, state_name)
   - on_state_exit(doc_type, doc_id, state_name)
   - Use hooks to update legacy status fields
